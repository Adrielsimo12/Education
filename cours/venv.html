ENV.01
Pourquoi utiliser .venv ?Isolation des dépendances et reproductibilité

Le problème sans environnements virtuels

Imaginons que vous ayez deux projets Python :

  • Projet A nécessite Flask 2.3.0
  • Projet B nécessite Flask 3.0.0

Si vous installez Flask globalement, les deux projets utiliseront la même version, ce qui peut créer des conflits.

Problème : Installer des packages Python globalement pollue le système et rend difficile le partage de projets avec d'autres développeurs.

La solution : environnements virtuels

Un environnement virtuel est un dossier isolé contenant une copie de Python et ses propres packages. Chaque projet a son propre environnement, évitant les conflits.

Isolation
Chaque projet a ses propres dépendances
Flask 2.3 pour projet A, Flask 3.0 pour projet B
Reproductibilité
Partage facile avec requirements.txt
Tous les collaborateurs ont les mêmes versions
Sécurité
Ne pas polluer le Python système
Évite les bugs système
Points clés
  • Toujours créer un environnement virtuel par projet Python
  • Le dossier .venv (ou venv) contient l'environnement
  • Activer l'environnement avant d'installer des packages ou d'exécuter du code
  • Ajouter .venv/ dans .gitignore (ne pas versionner l'environnement)
ENV.02
Créer un environnement virtuelvenv, activation et désactivation

Créer un environnement virtuel

Bash 
# Se placer dans le dossier du projet
cd mon-projet-python

# Créer un environnement virtuel nommé .venv
python3 -m venv .venv

# Sur Windows (si python3 n'existe pas)
python -m venv .venv

Cela crée un dossier .venv (le point au début le rend caché sur Linux/macOS) contenant :

  • bin/ (ou Scripts/ sur Windows) : scripts d'activation et exécutables
  • lib/ : packages Python installés
  • include/ : headers C (pour packages natifs)

Activer l'environnement virtuel

Système Commande d'activation
Linux/macOS source .venv/bin/activate
Windows (cmd) .venv\Scripts\activate.bat
Windows (PowerShell) .venv\Scripts\Activate.ps1

Une fois activé, vous verrez (.venv) au début de votre ligne de commande :

Bash 
# Avant activation
user@laptop:~/mon-projet$ 

# Après activation
(.venv) user@laptop:~/mon-projet$

Désactiver l'environnement

Bash 
# Désactiver l'environnement (sur tous les systèmes)
deactivate
VS Code : Si vous ouvrez un dossier contenant un .venv, VS Code le détecte automatiquement et propose de l'utiliser. Le terminal intégré l'activera automatiquement !
ENV.03
Gestion des packagespip install, freeze, requirements.txt

Installer des packages avec pip

Bash 
# TOUJOURS activer l'environnement d'abord
source .venv/bin/activate

# Installer un package
pip install flask

# Installer plusieurs packages
pip install flask flask-cors requests

# Installer une version spécifique
pip install flask==3.0.0

# Mettre à jour pip (recommandé)
pip install --upgrade pip

Lister les packages installés

Bash 
# Voir tous les packages installés
pip list

# Voir les informations d'un package
pip show flask

Créer requirements.txt

Le fichier requirements.txt liste toutes les dépendances du projet. Il permet à d'autres développeurs de recréer exactement le même environnement.

Bash 
# Générer requirements.txt avec les versions exactes
pip freeze > requirements.txt

# Contenu exemple de requirements.txt :
# Flask==3.0.0
# Flask-CORS==4.0.0
# requests==2.31.0

Installer depuis requirements.txt

Lorsqu'un nouveau développeur clone le projet, il peut recréer l'environnement :

Bash 
# 1. Créer l'environnement virtuel
python3 -m venv .venv

# 2. Activer
source .venv/bin/activate

# 3. Installer toutes les dépendances
pip install -r requirements.txt
Bonnes pratiques : Mettez à jour requirements.txt après chaque nouvelle installation avec pip freeze > requirements.txt.
ENV.04
Workflow completDe la création au déploiement

Démarrer un nouveau projet Python

Bash 
# 1. Créer un dossier projet
mkdir mon-api-flask
cd mon-api-flask

# 2. Créer l'environnement virtuel
python3 -m venv .venv

# 3. Activer
source .venv/bin/activate

# 4. Installer les packages nécessaires
pip install flask flask-cors

# 5. Créer requirements.txt
pip freeze > requirements.txt

# 6. Initialiser Git
git init

# 7. Créer .gitignore
echo ".venv/" > .gitignore
echo "__pycache__/" >> .gitignore
echo ".env" >> .gitignore

# 8. Premier commit
git add .
git commit -m "Initial commit"

Travailler sur un projet existant

Bash 
# 1. Cloner le projet depuis GitHub
git clone https://github.com/user/mon-projet.git
cd mon-projet

# 2. Créer l'environnement virtuel
python3 -m venv .venv

# 3. Activer
source .venv/bin/activate

# 4. Installer les dépendances depuis requirements.txt
pip install -r requirements.txt

# 5. Lancer l'application
python app.py

Exercices & Travaux PratiquesGestion des environnements Python et des dépendances

La gestion correcte des environnements Python est essentielle pour éviter les conflits de dépendances et garantir la reproductibilité de vos projets.

📝 Exercice 1 — Gestion de base des venv Facile

  1. Créez deux projets Python avec des versions différentes d'une même bibliothèque (ex: Flask 2.x dans le projet A, Flask 3.x dans le projet B). Démontrez qu'ils coexistent sans conflit.
  2. Exportez les dépendances avec pip freeze > requirements.txt. Créez un nouveau venv et installez toutes les dépendances depuis ce fichier. Vérifiez l'identité des environnements.
  3. Utilisez pip list --outdated pour identifier les packages obsolètes. Mettez-les à jour sélectivement et testez que l'application fonctionne encore.

📝 Exercice 2 — Outils avancés (poetry, pipenv) Moyen

  1. Comparez venv + pip, pipenv et poetry en créant le même projet avec chacun des trois. Notez les différences en termes de workflow, fichiers créés et commandes.
  2. Avec Poetry : initialisez un projet (poetry init), ajoutez des dépendances de production et de développement, publiez le pyproject.toml sur GitHub. Faites cloner et installer par un camarade avec poetry install.
  3. Créez un Makefile avec les cibles setup, run, test et clean pour automatiser la gestion du projet.

🔬 TP — Pipeline de développement Python professionnel Avancé — 2h

  1. Structure : Créez un projet Python bien structuré avec src/, tests/, docs/, .github/workflows/.
  2. Qualité : Configurer black (formatage), flake8 (lint), mypy (types statiques), pytest (tests unitaires).
  3. Pre-commit hooks : Installer pre-commit et configurer les hooks qui vérifient automatiquement la qualité du code avant chaque commit.
  4. CI/CD : Créer un workflow GitHub Actions qui installe les dépendances, lance les tests et vérifie le formatage à chaque push.
