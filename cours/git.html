GIT.01
Introduction à GitPourquoi utiliser Git ?

Qu'est-ce que Git ?

Git est un système de contrôle de version qui permet de :

  • Suivre l'historique de chaque modification du code
  • Revenir à une version antérieure en cas d'erreur
  • Collaborer efficacement avec d'autres développeurs
  • Gérer plusieurs versions (branches) d'un projet en parallèle
Git ≠ GitHub : Git est l'outil de gestion de version local. GitHub est une plateforme en ligne pour héberger et partager des dépôts Git (alternatives : GitLab, Bitbucket).

Installation de Git

Système Installation
Windows Télécharger depuis git-scm.com
macOS brew install git (avec Homebrew)
Linux (Ubuntu/Debian) sudo apt install git

Vérifier l'installation :

Bash 
git --version
# Résultat : git version 2.40.1

Configuration initiale

Bash 
# Configurer votre identité
git config --global user.name "Votre Nom"
git config --global user.email "votre.email@example.com"

# Vérifier la configuration
git config --list
GIT.02
Commandes de baseinit, add, commit, status, log

Créer un dépôt Git (repository)

Bash 
# Se placer dans le dossier du projet
cd mon-projet

# Initialiser Git
git init
# Crée un dossier caché .git qui contient l'historique

Workflow Git de base

1. Travailler
Modifier/créer des fichiers dans votre projet
index.html, style.css, etc.
2. Ajouter (stage)
Préparer les fichiers pour le commit
git add .
3. Valider (commit)
Sauvegarder les changements dans l'historique
git commit -m "message"

Commandes essentielles

Bash 
# Voir l'état des fichiers (modifiés, non suivis, etc.)
git status

# Ajouter un fichier spécifique à la zone de staging
git add index.html

# Ajouter TOUS les fichiers modifiés
git add .

# Valider les changements avec un message descriptif
git commit -m "Ajout de la page d'accueil"

# Voir l'historique des commits
git log

# Historique condensé (une ligne par commit)
git log --oneline
Bonnes pratiques : Écrivez des messages de commit clairs et descriptifs. Ex : "Ajout de la fonctionnalité de recherche" plutôt que "Update".
GIT.03
BranchesTravailler sur plusieurs versions en parallèle

Qu'est-ce qu'une branche ?

Une branche est une version parallèle de votre projet. Par défaut, vous êtes sur la branche main (ou master). Créer une branche permet de développer une fonctionnalité sans affecter le code principal.

Exemple : Vous travaillez sur une nouvelle fonctionnalité "login". Créez une branche feature-login, développez-y votre code, testez-le, puis fusionnez-la dans main quand c'est prêt.

Commandes de gestion des branches

Bash 
# Voir toutes les branches
git branch

# Créer une nouvelle branche
git branch feature-login

# Changer de branche (basculer sur feature-login)
git checkout feature-login

# Créer ET basculer sur une nouvelle branche (raccourci)
git checkout -b feature-contact

# Fusionner une branche dans la branche actuelle
git checkout main
git merge feature-login

# Supprimer une branche
git branch -d feature-login

Workflow avec branches

Étape Commande
1. Créer une branche git checkout -b nouvelle-fonctionnalite
2. Travailler sur la branche Modifier les fichiers, git add ., git commit -m "..."
3. Revenir sur main git checkout main
4. Fusionner la branche git merge nouvelle-fonctionnalite
5. Supprimer la branche git branch -d nouvelle-fonctionnalite
GH.01
GitHubHéberger et partager vos projets

Créer un compte GitHub

Rendez-vous sur github.com et créez un compte gratuit. GitHub vous permet d'héberger vos projets Git en ligne, de collaborer avec d'autres développeurs et de contribuer à des projets open-source.

Créer un dépôt (repository) sur GitHub

  1. Cliquer sur le bouton "New repository" sur GitHub
  2. Donner un nom au dépôt (ex: mon-site-web)
  3. Choisir "Public" ou "Private"
  4. Cliquer sur "Create repository"

Lier votre projet local à GitHub

Bash 
# Ajouter l'URL du dépôt distant (remote)
git remote add origin https://github.com/votre-nom/mon-site-web.git

# Vérifier les dépôts distants
git remote -v

# Envoyer (push) votre code sur GitHub (branche main)
git push -u origin main

# Les prochains push seront plus simples :
git push

Cloner un dépôt existant

Bash 
# Télécharger un projet depuis GitHub
git clone https://github.com/utilisateur/nom-du-projet.git

# Se déplacer dans le dossier cloné
cd nom-du-projet
SSH vs HTTPS : Pour éviter de taper votre mot de passe à chaque push, configurez une clé SSH. Voir la documentation GitHub : docs.github.com/authentication/connecting-to-github-with-ssh
GH.02
CollaborationPull, fetch, pull requests

Récupérer les modifications (pull)

Si quelqu'un a modifié le code sur GitHub (ou si vous avez travaillé depuis un autre ordinateur), utilisez git pull pour récupérer les dernières modifications.

Bash 
# Récupérer et fusionner les modifications de GitHub
git pull origin main

# Raccourci (si branche tracking configurée)
git pull

Pull Requests (PR)

Une Pull Request est une demande pour fusionner une branche dans une autre (souvent de feature-X vers main). C'est le mécanisme standard pour contribuer à un projet sur GitHub.

  1. Créer une branche : git checkout -b nouvelle-feature
  2. Faire des commits sur cette branche
  3. Pousser la branche sur GitHub : git push origin nouvelle-feature
  4. Sur GitHub, cliquer sur "Compare & pull request"
  5. Ajouter une description et cliquer sur "Create pull request"
  6. Les collaborateurs peuvent relire le code, commenter, puis fusionner (merge)

Résoudre les conflits

Parfois, Git ne peut pas fusionner automatiquement (conflit). Vous devez éditer manuellement les fichiers concernés, retirer les marqueurs de conflit (<<<<<<<, =======, >>>>>>>), puis faire un commit.

Bash 
# Si conflit lors d'un merge
git merge feature-X
# Git indique les fichiers en conflit

# Ouvrir les fichiers en conflit dans VS Code
# Résoudre manuellement, puis :
git add .
git commit -m "Résolution du conflit"
TIPS
Bonnes pratiques.gitignore, commits fréquents, messages clairs

Fichier .gitignore

Le fichier .gitignore permet d'exclure certains fichiers/dossiers du suivi Git (node_modules, fichiers de config sensibles, fichiers temporaires, etc.).

.gitignore 
# Dossiers
node_modules/
__pycache__/
venv/
.venv/

# Fichiers sensibles
.env
config.json

# Fichiers OS
.DS_Store
Thumbs.db

# IDE
.vscode/
.idea/
Conseils d'expert
  • Commits fréquents : Faites des commits réguliers avec des messages clairs
  • Une fonctionnalité = une branche : Ne travaillez jamais directement sur main
  • Pull avant push : Toujours faire git pull avant git push pour éviter les conflits
  • README.md : Ajoutez un fichier README pour documenter votre projet
  • Sauvegardes régulières : Poussez votre code sur GitHub régulièrement
  • Ne jamais commit de secrets : Mots de passe, clés API → utilisez .gitignore et variables d'environnement

Exercices & Travaux PratiquesVersioning, branches, merge, rebase et collaboration GitHub

Git est un outil fondamental du développeur moderne. Ces exercices vous guident de l'initialisation d'un dépôt jusqu'à la gestion de conflits complexes et la collaboration en équipe via GitHub.

Exercices de compréhension

📝 Exercice 1 — Commandes Git fondamentales Facile

  1. Initialisez un dépôt Git local. Créez 3 fichiers, faites un premier commit. Modifiez un fichier, utilisez git diff pour voir les changements, puis committez.
  2. Consultez git log --oneline --graph --all et expliquez chaque partie de l'affichage.
  3. Créez un fichier .gitignore qui ignore : tous les fichiers .env, le dossier node_modules/, les fichiers .pyc et le dossier __pycache__/.
  4. Utilisez git stash pour sauvegarder des modifications en cours sans committer. Récupérez-les avec git stash pop. Quand utiliser stash ?

📝 Exercice 2 — Branches et stratégies de merge Moyen

  1. Créez une branche feature/login depuis main. Faites 3 commits dessus. Revenez sur main, faites 1 commit. Mergez la feature avec --no-ff et observez le graphe.
  2. Créez intentionnellement un conflit de merge : modifiez la même ligne du même fichier sur deux branches différentes. Résolvez le conflit manuellement dans VS Code. Committez la résolution.
  3. Utilisez git rebase sur la même situation. Comparez le graphe obtenu avec le merge. Quelle est la règle d'or du rebase ?
  4. Utilisez git cherry-pick HASH pour appliquer un seul commit d'une branche sur une autre. Quand est-ce utile ?

Travaux Pratiques

🔬 TP — Workflow Git en équipe (simulation) Avancé — 2h

Objectif : Simuler un workflow Git Flow complet avec GitHub.

  1. Setup : Créez un dépôt GitHub. Invitez un camarade comme collaborateur. Définissez des règles de protection sur main (merge uniquement via PR approuvée).
  2. Git Flow : Créer les branches main (production), develop (intégration), feature/X (fonctionnalités).
  3. Pull Requests : Chaque fonctionnalité est développée sur sa branche, puis soumise en PR vers develop. L'autre étudiant review et approuve ou demande des modifications.
  4. Issues : Créez des issues GitHub pour chaque tâche. Liez les commits aux issues avec git commit -m "feat: add login page (closes #5)".
  5. CI : Ajoutez un fichier .github/workflows/ci.yml qui lance automatiquement les tests à chaque push.

❓ Quiz de révision — Git

  1. Quelle est la différence entre git fetch et git pull ?
  2. Expliquez ce que fait git reset --hard HEAD~2. Quelle est la différence avec --soft et --mixed ?
  3. Qu'est-ce qu'un dépôt "bare" (git init --bare) ? Dans quel contexte est-il utilisé ?
  4. Expliquez la différence entre git merge, git rebase et git squash. Quand utiliser chacun ?
  5. Qu'est-ce qu'un tag Git ? Quelle est la différence entre un tag léger et un tag annoté ?
